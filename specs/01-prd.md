# 01 - Product Requirements Document (PRD)

> SDD 单源真相之一。本文定义"做什么"和"为什么"，**不**包含"怎么做"（架构见 [02-architecture.md](02-architecture.md)）。

## 1. 目标用户

**个人用户**（美食爱好者/学生/工程师），使用本工具：
- 在"不知道吃什么"时获得个性化推荐
- 探索不同菜系/餐厅/吃法
- 学习 Agent 工程化的参考实现

**次要用户**：Agent 开发者，学习多 Agent 编排模式。

## 2. 核心场景

### 场景 A：日常决策
> "今天下雨，心情不太好，预算 100，想一个人吃点热乎的"

**期望**：考虑天气（下雨→热食）、心情（治愈系）、预算（≤100）、场景（独处），推 2-3 个候选 + 推荐理由。

### 场景 B：约会筹备
> "这周六想约女朋友吃个日料，不要太吵，人均 300 左右"

**期望**：场景（约会/不要吵）、菜系（日料）、价格（300）、位置（隐含 → 询问），输出有情调的日料店。

### 场景 C：快餐/小吃
> "中午懒得做饭，来个麦当劳或者沙县吧"

**期望**：识别"快餐"意图，并行咨询西式快餐 + 中式快餐专家。

### 场景 D：解决选择困难
> "想吃辣的，但不知道川菜还是湘菜"

**期望**：并行咨询两个菜系专家，对比后给推荐。

### 场景 E：连续对话（记忆）
> 第一次："我海鲜过敏"
> 第二次："推荐点菜" → 自动避开海鲜

## 3. 非目标

- ❌ 商业化订餐/支付
- ❌ 真实餐厅实时数据（用 mock + MCP 抽象）
- ❌ 多用户系统（单机本地）
- ❌ 移动端 App
- ❌ 餐厅评分爬虫

## 4. 验收标准（Definition of Done）

### MVP (Phase 1-3)
- [ ] CLI 能交互"我想吃辣的"→ 推荐川菜
- [ ] 8 维分析器全部能跑
- [ ] 14 个菜系专家可加载

### 完整版 (Phase 4-8)
- [ ] Gradio Web Demo 可访问
- [ ] 第二次会话记住"海鲜过敏"
- [ ] MCP 接入天气后，下雨时推荐会变
- [ ] 工具失败自动降级，UI 不崩
- [ ] 测试覆盖率 ≥ 80%
- [ ] README 演示新增菜系 < 5 分钟

## 5. 成功指标

| 指标 | 目标 |
|---|---|
| 推荐"靠谱率" | 用户主观 ≥ 70% 觉得"确实想吃" |
| 响应时间 | 端到端 < 8s（单轮） |
| Token 消耗 | 单次推荐 < 8K tokens |
| 工具成功率 | ≥ 95% (含重试后) |
| 测试覆盖 | ≥ 80% |

## 6. 优先级

| P0 | P1 | P2 |
|---|---|---|
| Master + 1 菜系 E2E | 14 菜系 + 8 维 | 真实 MCP API |
| Mock 数据 | 短期/长期记忆 | 推荐历史学习 |
| CLI 跑通 | Gradio Demo | A/B 测试框架 |
| Retry 兜底 | Tracing + 缓存 | 多用户支持 |

## 7. 用户故事（精简版）

1. 作为用户，**我想要**说"想吃辣的"，**以便**得到川菜/湘菜推荐。
2. 作为用户，**我想要**提到"下雨"，**以便**得到热乎乎的菜。
3. 作为用户，**我想要**说"约会"，**以便**得到有情调的餐厅。
4. 作为用户，**我想要**说"海鲜过敏"，**以便**后续推荐自动避开。
5. 作为开发者，**我想要**在 `cuisines.yaml` 加 4 行，**以便**新增菜系。
6. 作为开发者，**我想要** mock LLM 测试，**以便**不消耗 token。
